# ADR: コールバックエンドポイントのエラーレスポンスを 403 に統一する

## Status

Accepted

## Context

OIDC Authorization Code Flow のコールバックエンドポイント（`GET /callback`）では、以下のような複数の失敗ケースが存在する。

| ケース | 発生元 |
|---|---|
| OP が `error` パラメータを返す（`access_denied`, `server_error` 等） | OP |
| `state` 検証失敗（不一致、TTL 切れ） | RP |
| `code` / `state` パラメータ欠落 | リクエスト不正 |
| `id_token` 署名検証失敗 | RP |
| `id_token` Claims 検証失敗（`iss`, `aud`, `exp` 等） | RP |

これらに対して HTTP ステータスコードをどう割り当てるかを検討した。

### 検討した選択肢

**A. 細かく分割する**（400 / 401 / 403 / 502 / 503）

- `access_denied` → 403
- `server_error` / `temporarily_unavailable` → 502 / 503
- `state` 不一致 → 400
- `id_token` 検証失敗 → 401

意味的には正確だが、フロントエンドがステータスコードに応じた UI 分岐を必要としない限り過剰に複雑。

**B. 403 に統一する**

RP の視点では「このリクエストではリソースへのアクセスを許可できない」という事実が共通している。理由の細分化はブラウザには不要であり、詳細はサーバーログで追う。

## Decision

**B. 403 に統一する** を採用する。

### 根拠

- **oauth2-proxy（CNCF プロジェクト）の実装と一致する**
  oauth2-proxy は OP から `error` パラメータが返された場合、種類を問わず `http.StatusForbidden`（403）を返している。プロダクション実績のある実装に倣う。

- **フロントエンドへの影響が最小**
  フロントエンドが必要とする情報は「ログインできなかった」という事実であり、理由の細分化は現時点で要求されていない。詳細が必要な場合はレスポンスボディの `error` フィールドで提供できる。

- **シンプルさ優先**
  `server_error` / `temporarily_unavailable` を 502/503 で返す方が意味的には正確だが、OP 障害の監視はサーバーログ・メトリクスで行うべきであり、HTTP ステータスコードに持たせる必要はない。

## Consequences

- コールバックの失敗はすべて 403 で返る。フロントエンドは 403 を「ログイン失敗」として扱えばよい。
- 失敗の詳細（`access_denied` なのか `state` 不一致なのか等）はレスポンスボディおよびサーバーログで確認する。
- 将来フロントエンドが失敗理由に応じた UI 分岐を必要とする場合は、ステータスコードの細分化ではなくレスポンスボディの `error` フィールドを拡充する方向で対応する。
